# 機能追加ワークフロー

`$ARGUMENTS` を実装する。以下のステップを順番に実行すること。

## 1. 調査

- 既存コードを読み、関連するファイル・パターン・テストを把握する
- `src/routes/` のルーティングパターン、`src/services/` のサービス層、`src/storage/` のストレージ層を確認
- 既存テスト (`test/`) の書き方を確認してスタイルを合わせる

## 2. 実装

- 既存のコードスタイル・パターンに従って実装する
- 必要に応じてバリデーション（`isValidSlug` 等）を適用する
- 認証が必要なエンドポイントは `apiRoute` 配下に置く（`bearerAuth` ミドルウェアが自動適用される）

## 3. テスト

- `test/` 配下に既存のテストファイルがあればそこに追加、なければ新規作成
- 正常系・異常系（バリデーションエラー、404、認証なし等）をカバーする
- `bun test` で全テストが通ることを確認する

## 4. ドキュメント更新

以下のドキュメントを変更内容に合わせて更新する。更新が不要なものはスキップしてよい。

- **`docs/api.md`** — API エンドポイントを追加・変更した場合、リファレンスに追記する。既存セクションの書式（テーブル、レスポンス例）に合わせること
- **`docs/guide/architecture.md`** — アーキテクチャに影響する変更（新しいレイヤー、バインディング追加等）があれば更新
- **`docs/guide/security.md`** — セキュリティに関わる変更（認証、CORS、ヘッダー等）があれば更新
- **`docs/mcp.md`** — MCP サーバー関連の変更があれば更新

## 5. コミット & PR 作成

- main から新しいブランチを作成する（ブランチ名: `feat/<簡潔な説明>`）
- 変更をコミットする（コミットメッセージは日本語で、conventional commits 形式）
- `gh pr create` で PR を作成する。PR のタイトルは英語、本文は日本語で Summary と Test plan を含める
